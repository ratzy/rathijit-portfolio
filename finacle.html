The most product-shaped thing I've built

A design system isn't a project. It's a product — with users, a roadmap, a versioning story and a support burden — and its users are the hardest audience you'll ever design for, because they're designers and engineers who already have opinions.

This is the one I'd point to first.

The problem

Finacle is cloud-based online banking — onboarding, sales, service, engagement — with tailored UI for retail, SME and corporate customers. Multiple product tracks, each with its own design language.

The same dropdown behaved differently in three places. The same action lived in three positions. Customers moving between products experienced them as three different banks.

How do we unify every track under one language — and raise usability with accessibility built in, not bolted on?

The constraint that shaped everything: this had to be technology-agnostic. Tracks ran on different frameworks. A system that only worked in one stack would fracture the moment it met reality.

Why this problem is harder than it looks

You cannot mandate a design system into existence. Every track had shipped products, deadlines, and engineers who had already solved these problems their own way. A system imposed from outside gets ignored politely and routed around quietly.

So the real problem was never components. It was adoption. The design work had to be so obviously better — and so obviously easier — that using it beat not using it. That reframing drove every decision below.

The approach — a loop, not a line

Gather requirements → Analysis → Brainstorm → Research → Ruleset & Documentation → Accessibility → Ideate → Usability testing → Functional testing → Deliver / Release → Feedback.

Note the shape: feedback returns to the top. A design system that ships once is a style guide. One that ships continuously is a product.

Building from the atom up

Atoms → molecules. A circle, a square, a text block — composed into a message component.

Anatomy of one component

I'll show one, because one is enough to prove the rest. The dropdown.

Specification — a contract, not a suggestion

Overview: what it is, when it activates, how selection works. Anatomy: label, then options — single/multi select, and the active dropdown icon. Specification: collapsed and expanded states, measured. 8px, 36px, 24px, 30px. Grid widths min 04, max 06.

Usage — the part most systems skip

Telling designers where a component fits is easy. Telling them where not to use it is what makes a system trustworthy.

Accessibility, designed in

Not a compliance pass at the end — a state in the component spec. Every dropdown state documented, keyboard-first, with the colour contract to match: focus outline, error, disabled fill, read-only. An engineer implementing this cannot accidentally build an inaccessible dropdown — the accessible version is the only version documented. Level AA of WCAG 2.1, across the system.

Proving decisions with users

We didn't argue about approaches. We tested them. Two table-filter approaches — combined vs individual filters. 7 participants, novice through expert. The hypothesis: fewer clicks per user, but is a single-filter scenario better with fewer columns?

Approach 1 — combined4.21 min100%1

Both completed at 100% — so completion couldn't decide it. Time and preference could, and they agreed: Approach 2, by 0.36 minutes and 6 votes to 1. Run over Microsoft Teams, measuring timing and functionality while probing workflow comprehension.

Why I'm showing you this: it's the closest thing in my portfolio to a real product decision made on real evidence, with a real trade-off, verified with real users. Not a preference. A finding.

Research grounded in the actual job

Before components, the use case: a bank teller trying to find out whether the person in front of them is already a customer. They need fast, accurate search; a clear interface; comprehensive profiles in one place; and an efficient new-account path. Their pain: slow, cumbersome search, inaccurate results, an unintuitive CRM, and no complete picture of the customer.

The pattern, applied

Enquiry → results → refined search. Built from the system's own atoms.

How I led it

  • Strategy & execution. Built the strategy, ran workshops, analysed research, usability testing, mid- and high-fidelity wireframes; walkthroughs to client and execution teams.
  • Delivered components, not decks. Production-ready assets and code to engineering and framework teams — cross-platform consistency guaranteed.
  • Design reviews. Facilitated feedback, synthesised insights, folded improvements back into the system.
  • Mentored design & dev teams. Deep understanding of the system so it got used correctly, not just used.
  • Maintenance strategy. A plan for ongoing upkeep with regular updates — because a system without a maintenance plan has an expiry date.
  • Culture. Weekly team activities, monthly appreciation for star performers.

What changed

Four design languages became one. Accessibility stopped being a review gate and became the default path. Engineering received code, not interpretation. And the system was built to keep evolving — testing and refinement led continuously, so it adapts as user needs and technology move.

What I'd do differently

I measured the system's health, not the product's. +70% consistency is real — but it's an internal metric. It tells you the system worked, not that customers had a better time at the bank.

I never instrumented the thing that mattered: did teller search get faster in production? Did account-setup time drop? Did the tracks that adopted earliest ship better outcomes? That data existed. I didn't ask for it.

I'd also track adoption as a funnel — which tracks adopted, how fast, where they resisted, what they forked and why. Every fork is a bug report about the system, and I was treating them as compliance failures rather than signal. The system was rigorous. The measurement of the system wasn't. That's the discipline I'm carrying into a product role — and exactly why I want one.
